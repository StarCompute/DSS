📡一、HTTP 客户端统一封装

核心原则：项目中禁止直接使用原生 fetch/XMLHttpRequest。必须通过统一封装的 HTTP 客户端发起所有请求， 以确保：统一的认证令牌注入、统一的错误拦截、统一的请求/响应日志、统一的超时和重试策略。

axios 统一封装模板

utils/request.js — axios 统一封装
// ===== utils/request.js =====
import axios from 'axios';
import { getToken, refreshToken, redirectToLogin } from './auth';
import { ElMessage } from 'element-plus'; // 或其他UI库

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE || '/api',
  timeout: 15000,                        // 15秒超时
  headers: { 'Content-Type': 'application/json' }
});

// ===== 请求拦截器 =====
request.interceptors.request.use(config => {
  const token = getToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  // 注入追踪ID
  config.headers['X-Request-ID'] = crypto.randomUUID();
  return config;
}, error => Promise.reject(error));

// ===== 响应拦截器 =====
request.interceptors.response.use(
  response => {
    const { code, data, message } = response.data;
    if (code === 0) return data;           // 成功 → 直接返回 data
    if (code === 401) {                    // Token过期
      return refreshToken().then(() => request(response.config));
    }
    ElMessage.error(message || '请求失败');
    return Promise.reject(new Error(message));
  },
  error => {
    if (error.response) {
      const { status } = error.response;
      if (status === 401) redirectToLogin();
      if (status === 403) ElMessage.error('无权限访问');
      if (status >= 500) ElMessage.error('服务器异常，请稍后重试');
    } else if (error.code === 'ECONNABORTED') {
      ElMessage.error('请求超时，请检查网络');
    } else {
      ElMessage.error('网络异常，请检查连接');
    }
    return Promise.reject(error);
  }
);

export default request;

业务错误码体系

code 范围含义示例
0成功正常响应
1000-1999参数校验错误1001=必填项缺失，1002=格式不正确
2000-2999认证授权错误2001=未登录，2002=Token过期，2003=无权限
3000-3999业务规则错误3001=库存不足，3002=订单状态不允许
4000-4999资源不存在4001=用户不存在，4002=文件不存在
9000-9999系统内部错误9001=数据库异常，9002=第三方服务异常

HTTP 状态码使用规范

状态码使用场景说明
200GET/PUT 成功、POST 创建成功最常用
201POST 创建资源成功推荐用于创建操作，返回新资源URL
204DELETE 成功（无响应体）删除操作推荐
400请求参数错误（校验失败）携带具体校验错误信息
401未认证（未登录/Token无效）前端拦截器自动跳转登录页
403无权限（已登录但权限不足）与401区别：403=身份已知但无权限
404资源不存在
422语义错误（请求格式正确但业务逻辑不通过）推荐用于复杂业务校验
429请求频率超限配合 Retry-After 头
500服务器内部错误不暴露具体错误信息给客户端
502网关错误（上游服务不可用）
503服务暂不可用（维护/过载）

🏷️四、请求头管理规范

Header必选值示例说明
Authorization ✅ Bearer eyJhbGciOi... JWT 认证令牌，由拦截器自动注入
Content-Type ✅ application/json 请求体类型
X-Request-ID 推荐 550e8400-e29b-... 请求追踪ID（UUID v4），贯穿前后端日志
X-Client-Version 推荐 2.1.0 前端版本号，用于灰度发布/兼容性判断
Accept-Language 可选 zh-CN 国际化语言偏好
⚠️ 禁止：不要在请求头中传递业务参数（如用户ID、部门ID等）。业务参数应放在 URL Path/Query 或 Request Body 中。请求头仅用于协议级元数据。

🛡️五、全局错误拦截与处理

错误分类与处理策略

错误类型识别方式前端处理用户提示
网络断开 error.message === 'Network Error' 全局提示 + 暂停后续请求 "网络连接失败，请检查网络"
请求超时 error.code === 'ECONNABORTED' 提示 + 可重试 "请求超时，请重试"
401 未认证 status === 401 尝试 Refresh Token → 失败则跳登录 无提示（静默跳转）
403 无权限 status === 403 提示 + 可跳转首页 "您没有权限执行此操作"
404 不存在 status === 404 按接口决定：提示或跳404页 "请求的资源不存在"
422 校验失败 status === 422 将字段错误映射到表单控件 各字段下方显示错误
5xx 服务端错误 status >= 500 全局提示 + 不重试（避免雪崩） "服务器繁忙，请稍后重试"

全局 Loading 管理

请求计数器 — 全局 Loading 管理
// ===== 请求计数器模式（避免多个请求闪烁Loading） =====
let loadingCount = 0;

request.interceptors.request.use(config => {
  if (config.showLoading !== false) {
    loadingCount++;
    if (loadingCount === 1) showGlobalLoading();
  }
  return config;
});

request.interceptors.response.use(
  response => {
    if (response.config.showLoading !== false) {
      loadingCount--;
      if (loadingCount === 0) hideGlobalLoading();
    }
    return response;
  },
  error => {
    loadingCount--;
    if (loadingCount <= 0) { loadingCount = 0; hideGlobalLoading(); }
    return Promise.reject(error);
  }
);

步骤客户端服务端

2. 构建请求FormData 封装文件+附加参数—

前端上传封装

带进度的文件上传封装
// ===== 带进度的文件上传 =====
export function uploadFile(file, onProgress) {
  const formData = new FormData();
  formData.append('file', file);
  formData.append('category', 'document');  // 附加业务参数

  return request.post('/api/files/upload', formData, {
    headers: { 'Content-Type': 'multipart/form-data' },
    timeout: 120000,                          // 上传超时2分钟
    onUploadProgress: (progressEvent) => {
      const percent = Math.round(
        (progressEvent.loaded / progressEvent.total) * 100
      );
      onProgress?.(percent);
    }
  });
}

// ===== 使用示例 =====
const handleUpload = async (file) => {
  if (file.size > 10 * 1024 * 1024) {        // 10MB限制
    ElMessage.error('文件大小不能超过10MB');
    return;
  }
  try {
    const result = await uploadFile(file, (p) => {
      console.log(`上传进度: ${p}%`);
    });
    console.log('上传成功:', result.url);
  } catch (err) {
    ElMessage.error('上传失败');
  }
};

分片上传（大文件 > 50MB）

分片上传策略：大文件（>50MB）必须使用分片上传。 将文件切为 5MB/片，并发上传（最多3片并发），全部完成后调用合并接口。 断点续传：记录已上传分片序号，重传时跳过已完成分片。

🔄七、数据序列化约定

字段命名转换：camelCase ↔ snake_case

❌ 禁止：前后端字段名不一致
前端发送：{ userName: "张三" }
后端期望：{ user_name: "张三" }
→ 后端收到 undefined
✅ 方案：拦截器统一转换
前端内部：camelCase
发送请求前：转为 snake_case
收到响应后：转为 camelCase
对业务代码透明

日期时间序列化

场景格式示例
API 传输ISO 8601 (UTC)"2026-06-12T08:30:00Z"
仅日期ISO 8601 日期"2026-06-12"
前端展示本地化格式"2026年6月12日 16:30"
⚠️ 禁止：不要在 JSON 中使用时间戳数字（如 1718178600），人类不可读，调试困难。统一使用 ISO 8601 字符串。

重试策略

重试原则：仅对幂等请求（GET）和网络瞬时错误（Network Error、超时、5xx）进行自动重试。 POST/PUT/DELETE 等非幂等请求禁止自动重试，必须由用户手动触发。 重试采用指数退避：delay = min(1000 * 2^retryCount, 10000)，最多重试 3 次。

取消请求（AbortController）

搜索防抖 — AbortController 取消请求
// ===== 搜索防抖 + 取消上一次请求 =====
let abortController = null;

async function search(keyword) {
  // 取消上一次未完成的搜索请求
  if (abortController) abortController.abort();
  abortController = new AbortController();

  try {
    const result = await request.get('/api/search', {
      params: { keyword },
      signal: abortController.signal
    });
    return result;
  } catch (err) {
    if (err.name === 'CanceledError') return; // 被取消，静默处理
    throw err;
  }
}

✅数据交互 · 检查清单

✅ 是否使用了统一封装的 HTTP 客户端（禁止裸调 fetch/axios）？
✅ 请求/响应拦截器是否覆盖了Token注入、错误统一处理、Loading管理？
✅ API URL 是否遵循 RESTful 命名规范（名词复数、全小写短横线）？
✅ 所有接口响应是否遵循统一数据结构（code/message/data）？
✅ 业务错误码是否有明确的号段划分（1000/2000/3000...）？
✅ HTTP 状态码使用是否正确（201创建、204删除、422校验失败）？
✅ 401 响应是否触发Token 刷新（静默刷新→失败跳登录）？
✅ 文件上传是否有前端+后端双重校验（类型、大小、魔数）？
✅ 大文件（>50MB）是否使用分片上传？
✅ 日期时间是否统一使用 ISO 8601 字符串（禁止时间戳数字）？
✅ 前后端字段命名是否通过拦截器自动转换（camelCase ↔ snake_case）？
✅ POST/PUT/DELETE 是否禁止自动重试（避免重复操作）？
✅ 搜索/自动补全是否使用了AbortController 取消上一次请求？
✅ 是否注入了 X-Request-ID 用于全链路日志追踪？
