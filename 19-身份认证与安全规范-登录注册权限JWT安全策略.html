🔑一、认证机制选型

机制原理优点缺点推荐场景
JWT (推荐) 服务端签发签名的JSON令牌，客户端存储并在请求头携带 无状态、可扩展、适合分布式 无法主动失效（需配合黑名单） 中后台系统首选
Session + Cookie 服务端存储会话，客户端Cookie携带SessionID 可主动失效、成熟稳定 有状态、扩展需共享存储 传统单体应用
OAuth 2.0 / OIDC 第三方授权，获取Access Token访问资源 无需存储密码、支持SSO 配置复杂、依赖第三方 第三方登录、微服务SSO
推荐方案：中后台系统统一使用 JWT (Access Token + Refresh Token)。 Access Token 短期有效（15-30分钟），Refresh Token 长期有效（7-30天），兼顾安全与体验。

🎫二、JWT 设计规范

JWT Payload 标准结构

JWT Payload 标准结构
// ===== Access Token Payload =====
{
  "sub": "user_12345",        // 用户唯一标识（Subject）
  "name": "张三",             // 用户显示名
  "roles": ["admin","editor"],// 角色列表
  "perm": ["user:read",       // 权限编码列表（细粒度）
           "user:write",
           "report:export"],
  "iat": 1718178600,          // 签发时间（Unix时间戳）
  "exp": 1718179500,          // 过期时间（15分钟后）
  "iss": "system-name",       // 签发者
  "jti": "unique-token-id"    // Token唯一ID（用于黑名单）
}

// ===== Refresh Token Payload =====
{
  "sub": "user_12345",
  "type": "refresh",
  "iat": 1718178600,
  "exp": 1720770600,          // 30天后过期
  "jti": "unique-refresh-id"
}

Token 存储策略（前端）

存储位置安全性XSS风险CSRF风险推荐
localStorage ⚠️ 低 ❌ 高（JS可读） ✅ 安全 ❌ 不推荐
sessionStorage ⚠️ 低 ❌ 高（JS可读） ✅ 安全 ❌ 不推荐
httpOnly Cookie ✅ 高 ✅ 安全（JS不可读） ⚠️ 需配合CSRF防护 ✅ 推荐
内存变量（闭包） ✅ 高 ✅ 安全 ✅ 安全 ⚠️ 刷新丢失
🚫 绝对禁止：将 Token 存储在 localStorage 中！任何 XSS 攻击都可直接读取并窃取 Token。 推荐方案：Access Token 存内存（变量闭包），Refresh Token 存 httpOnly + Secure + SameSite=Strict Cookie。

🛂四、权限模型（RBAC）

核心原则：权限控制必须前后端双重校验。 前端控制 UI 显隐（提升体验），后端控制接口访问（安全保障）。后端权限校验不可被绕过。

三级权限粒度

粒度控制对象示例实现方式
页面级 菜单/路由可见性 无「用户管理」权限则不显示菜单 前端路由守卫 + 后端菜单接口过滤
按钮级 操作按钮显隐 无「删除」权限则隐藏删除按钮 前端 v-permission 指令 + 后端接口鉴权
数据级 数据可见范围 部门经理只看本部门数据 后端 SQL 注入数据范围条件

前端权限指令

Vue 3 权限指令与路由守卫
// ===== Vue 3 权限指令示例 =====
// v-permission="'user:delete'"
app.directive('permission', {
  mounted(el, binding) {
    const permissions = store.state.user.permissions;
    if (!permissions.includes(binding.value)) {
      el.parentNode?.removeChild(el);  // 无权限则移除DOM
    }
  }
});

// ===== 路由守卫 =====
router.beforeEach((to, from, next) => {
  const requiredPerm = to.meta.permission;
  if (requiredPerm && !hasPermission(requiredPerm)) {
    next('/403');  // 无权限跳转403页
  } else {
    next();
  }
});

后端权限中间件

PHP 权限中间件
// ===== PHP 中间件示例 =====
class PermissionMiddleware {
  public function handle($request, $next, $permission) {
    $user = auth()->user();
    if (!$user || !$user->hasPermission($permission)) {
      return response()->json([
        'code' => 2003,
        'message' => '无权限执行此操作'
      ], 403);
    }
    return $next($request);
  }
}
// 路由中使用
Route::delete('/api/users/{id}', [UserController::class, 'destroy'])
     ->middleware('permission:user:delete');

🛡️五、CSRF 防护规范

CSRF（跨站请求伪造）：攻击者诱导用户在已登录的站点执行非预期的操作。 如果使用 Cookie 存储认证信息，则必须启用 CSRF 防护。 如果使用 Authorization Header 传递 Token（非Cookie），则天然免疫 CSRF。

防护措施（至少启用两项）

措施优先级实现
SameSite Cookie 🔴 必选 Set-Cookie: token=xxx; SameSite=Strict; Secure; HttpOnly
CSRF Token 🟡 推荐 服务端生成随机Token，前端在请求头 X-CSRF-Token 中回传
Origin/Referer 校验 🟡 推荐 服务端校验请求来源是否在白名单中
自定义请求头 🟢 辅助 前端添加 X-Requested-With: XMLHttpRequest（触发浏览器预检）

🧹六、XSS 防护规范

防护矩阵

防护层措施负责方
输入层 所有用户输入进行白名单校验，拒绝非法字符 前端 + 后端
输出层 HTML上下文：&lt; &gt; &quot; &#x27; &amp; 转义 后端（模板引擎自动转义）

🔒七、密码安全策略

密码复杂度规范

参数最低要求推荐值
最小长度8 位12 位
字符类型至少包含 3 种：大写、小写、数字、特殊字符4 种全部
不允许连续字符（1234/abcd）、键盘序列（qwerty）、常见密码（password123）—
过期策略90 天强制修改—
历史限制不能与最近 5 次密码相同—

密码哈希算法

密码哈希算法 (bcrypt / argon2id)
// ===== bcrypt (推荐) =====
// PHP
$hash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 12]);
// cost=12 → 约0.3秒/次，平衡安全与性能

// Python
import bcrypt
hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

// Node.js
const bcrypt = require('bcrypt');
const hash = await bcrypt.hash(password, 12);

// ===== argon2id (更高安全需求) =====
// 抗GPU/ASIC破解，内存硬性消耗
// PHP: password_hash($p, PASSWORD_ARGON2ID)
🚫 绝对禁止：使用 MD5、SHA1、SHA256（未加盐） 存储密码！ 这些算法设计用于快速计算，极易被彩虹表或暴力破解。必须使用 bcrypt 或 argon2id。

🔏八、敏感数据保护

数据类型传输存储展示日志

手机号 HTTPS AES-256 加密 138****1234 脱敏后记录

银行卡号 HTTPS AES-256 加密 ****1234 不记录

⚠️ 日志安全红线：绝对不在日志中记录密码、Token、身份证号、银行卡号等敏感信息的明文。 日志系统必须具备自动脱敏能力，匹配敏感字段名（password/token/secret/idCard）自动替换为 ***。

✅身份认证与安全 · 检查清单

✅ 是否使用 JWT (Access + Refresh Token) 双令牌机制？
✅ Access Token 是否不超过30分钟过期？
✅ Token 是否存储在 httpOnly Cookie（而非 localStorage）？
✅ 是否实现了 Token 无感刷新（401→刷新→重放）？
✅ 密码是否使用 bcrypt (cost≥12) 哈希存储？
✅ 密码是否满足8位+3种字符类型的最低复杂度？
✅ 是否实现了登录失败锁定（5次/15分钟）？
✅ 权限控制是否前后端双重校验（前端UI显隐+后端接口鉴权）？
✅ 是否启用了 SameSite=Strict Cookie 防御 CSRF？
✅ 所有用户输入是否做了HTML转义（防XSS）？
✅ 富文本是否使用 DOMPurify 白名单过滤？
✅ 是否配置了 Content-Security-Policy 响应头？
✅ 敏感数据（手机号/身份证/银行卡）是否传输加密+存储加密+展示脱敏？
✅ 日志中是否不包含密码/Token/身份证等敏感信息？
✅ 是否记录了登录、权限变更、敏感操作的审计日志？
