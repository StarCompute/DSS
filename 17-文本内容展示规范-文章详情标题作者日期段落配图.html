🏗️一、内容页整体框架

核心原则：文本内容页（文章详情、公告详情、博客帖子、新闻页）必须遵循统一的五区结构。 每一区有明确的职责和视觉层级，不允许混用或省略关键区域。

五区结构模型

区域名称包含元素说明
① 标题区 H1 主标题、副标题（可选） 文章核心主题，一个页面只有一个 H1
② 元数据区 作者头像+姓名、发布日期、阅读量、分类标签 提供上下文信息，帮助读者判断内容可信度和时效
③ 正文区 段落、H2-H4 小标题、有序/无序列表、引用块、代码块、行内代码、链接、加粗/斜体 内容主体，所有文本样式在此区域统一
④ 标签/分类区 分类标签、关键词标签 便于内容归类和检索，位于正文之后
⑤ 附加区（可选） 版权声明、转载声明、相关文章推荐、评论区 增强互动和导航，按需展示

设计规范

在企业级中后台系统中，表单是用户与数据进行交互的核心媒介。一个设计良好的表单不仅能提升数据录入效率，还能显著降低操作错误率。本文将从信息架构、布局策略、交互反馈三个维度，系统性地阐述表单设计的最佳实践。

信息架构：先理清「为什么填」

在设计表单之前，首先要回答三个问题：用户为什么要填写这个表单？哪些字段是必填的？填写完成后会发生什么？将这些信息在表单顶部以简洁的方式告知用户，可以显著降低用户的认知负担。

💡 经验法则：每增加一个非必填字段，表单完成率下降约 3-5%。审慎评估每个字段的必要性。

在实际操作中，建议将表单字段分为三个优先级：核心必填（直接影响业务流转）、重要选填（提升数据质量但非必需）、补充信息（可后续完善）。

布局策略：纵向优先

研究表明，单列纵向布局的表单比多列布局的填写速度快 25%，因为用户不需要在列之间来回移动视线。

  • 标签统一置于输入框上方（顶部对齐），而非左侧
  • 相关联的字段使用分组标题分隔，每组不超过 5 个字段
  • 操作按钮（提交/取消）固定在页面底部或跟随滚动

如需了解更多细节，请参考 《表单全局规范》 中的完整说明。

🔤二、标题层级规范

核心原则：标题是内容骨架，必须形成清晰的层级树。每个页面有且仅有一个 H1， H2-H6 按嵌套顺序使用，不允许跳级（如 H2 后直接 H4）。

六级标题定义

级别字号字重行高颜色下间距使用场景
H1 26px 700 1.3 #111827 16px 文章主标题，每页仅一个
H2 22px 600 1.35 #111827 28px 一级章节标题（如"一、信息架构"）
H3 18px 600 1.4 #1F2937 20px 二级小节标题（如"1.1 字段分类"）
H4 16px 600 1.45 #374151 14px 三级细节标题（如"① 核心必填"）
H5 15px 500 1.5 #4B5563 10px 小标题/注意事项标题
H6 14px 500 1.5 #6B7280 8px 微型标题，极少使用

标题间距规则

关系间距值说明
标题上方（与前一内容）1.5× 标题下间距标题与上方内容距离 > 与下方内容距离，形成「归属感」
标题下方（与后一内容）见上表标题与其所属内容紧密相邻
标题与标题之间上间距 = 上一标题下间距 × 1.5同级标题间距一致，父子标题间距略小

📏三、段落与行距规范

核心原则：正文排版的目标是可读性最大化。字号、行高、段间距、阅读宽度四者必须协调配合。 研究表明，中文正文最佳阅读体验的参数范围为：字号 15-17px，行高 1.6-1.8，每行 35-45 个汉字。

正文排版核心参数

参数推荐值可接受范围依据
正文字号 16px 15-17px 小于 15px 增加阅读疲劳，大于 17px 降低信息密度
正文行高 1.75 1.6-1.8 中文笔画复杂，需要比英文更大的行高来保证字间透气性
段落间距 1.5em 1.2-2em 使用 em 单位相对于正文字号，响应式友好
最大阅读宽度 720px 680-760px 超出此宽度视线来回移动距离过大，降低阅读效率
正文颜色 #1F2937 — 不要用纯黑 #000，与白底对比过强导致眩光
段落对齐 text-align: justify（两端对齐） — 中文排版传统，两端对齐使段落左右边界整齐，视觉更规整；配合 word-break: break-word 避免英文长单词拉散文字
字间距 letter-spacing: 0（默认） 0-0.5px 中文默认字间距已满足阅读需求；正文场景不额外加 letter-spacing，标题可酌情加 0.5px 提升质感

首行缩进规范

核心原则：首行缩进是中文排版的重要传统，但在屏幕阅读场景下需分场景处理。 不是所有段落都需要缩进，也不是所有场景都适合缩进。
场景缩进值理由
长篇正文段落
（文章、博客、新闻正文）		 text-indent: 2em 中文传统排版规范：首行缩进两个汉字宽度（2em = 2×当前字号），帮助读者快速识别段落起始。这是中文印刷排版的标准实践。
首段（章节开头第一个段落） text-indent: 0（不缩进） 章节/标题后的第一个段落作为「引领段」，不缩进可保持版面整洁。标题与首段之间已有足够间距区分。
短段落（≤ 2 行） text-indent: 0（不缩进） 过渡性短段落、对话行、引言行等不应缩进，避免视觉碎片化。
列表项后首段 text-indent: 0（不缩进） 列表后的衔接段落通常延续上文语义，缩进会割裂上下文关联。
UI 界面段落
（卡片、弹窗、提示文案）		 text-indent: 0（不缩进） 界面空间有限，段落间距已足够区分；缩进会破坏 UI 对齐的几何美感。
首字下沉
（杂志/专题页风格）		 ::first-letter { font-size: 3em; float: left; } 适用于杂志风格、专题长文的首段首字，增强视觉吸引力。仅限创意场景，常规中后台内容不推荐。

首行缩进 CSS 实现

/* 标准正文段落：首行缩进 2 个汉字宽度 */
.article-body p {
  text-indent: 2em;
  text-align: justify;
  margin-bottom: 1.5em;
}
/* 首段不缩进（章节开头第一个 p） */
.article-body h2 + p,
.article-body h3 + p,
.article-body h4 + p,
.article-body hr + p,
.article-body blockquote + p {
  text-indent: 0;
}
/* 列表后首段不缩进 */
.article-body ul + p,
.article-body ol + p {
  text-indent: 0;
}
/* 首字下沉（杂志风格，按需启用） */
.article-body .drop-cap::first-letter {
  font-size: 3em;
  float: left;
  line-height: 1;
  margin-right: 6px;
  color: var(--primary);
}

👤四、作者/日期/元数据展示规范

核心原则：元数据区位于标题下方、正文上方，提供内容的可信度锚点。 作者、日期为必选字段，阅读量、分类标签为推荐字段。

元数据元素定义

元素必选样式说明
作者头像 ✅ 32×32px 圆形，如无头像用姓名首字替代 增加内容的人格化可信度
作者姓名 ✅ 14px / font-weight:500 / #1E293B 可附带职务（12px / #6B7280）
发布日期 ✅ 13px / #6B7280 / 格式 YYYY-MM-DD 统一日期格式，避免「3天前」等相对时间（内容页需要精确日期）
阅读量 推荐 13px / #6B7280 增强内容热度感知
分类标签 推荐 12px / 圆角 pill 标签 放在元数据行右侧，便于快速识别内容领域
预计阅读时长 可选 13px / #6B7280 如「阅读约 5 分钟」，提升用户预期管理

📋五、列表样式规范

有序列表 vs 无序列表

类型适用场景标记样式缩进项间距
有序列表 <ol> 步骤说明、排名、优先级排序 数字 + 点号（1. 2. 3.） padding-left: 24px margin-bottom: 8px

🔖十一、强调与语义化文本标签

核心原则：使用语义化标签表达内容含义，而非仅依赖视觉样式。语义标签对SEO、屏幕阅读器、内容聚合至关重要。 选择标签时优先考虑「这段文字在内容中扮演什么角色」，而非「这段文字应该长什么样」。

语义标签速查表

标签语义默认样式适用场景替换说明
<strong> 重要、紧急 font-weight: bold 关键结论、警告词、必须遵守的规则 ❌ <b> 语义上表达「重要性」，而不仅仅是视觉加粗
<em> 强调、重读 font-style: italic 句子中需要重读的词、对比强调 ❌ <i> 改变句子含义的重音位置。如「你做的？」vs「你做的？」
<b> 视觉加粗（无语义） font-weight: bold 产品名、关键词提取（不带重要性） — 仅在 HTML5 中重新定义为「样式偏移」标签，无语义权重
<i> 另类语音/语调 font-style: italic 外语短语、思想对白、分类学术语、船名 — HTML5 重新定义为「与主文本不同语调」的内容
<mark> 高亮标记 背景黄色 搜索结果匹配词、当前上下文相关的文本 ❌ <span> 表示与用户当前操作相关的文本，而非通用高亮
<del> / <ins> 删除 / 插入 删除线 / 下划线 文档修订记录、价格变更（原价→现价）、条款更新 ❌ <s> <s> 仅表示「不再准确/相关」，无编辑含义
<kbd> 键盘输入 等宽字体、小字号 快捷键说明（Ctrl+C）、命令行输入 ❌ <code> 行内代码用 <code>，键盘操作用 <kbd>
<abbr> 缩写/首字母缩略词 虚线下划线（部分浏览器） 首次出现的专业缩写，配合 title 属性提供全称 — 如 <abbr title="HyperText Markup Language">HTML</abbr>
<dl>/<dt>/<dd> 描述/定义列表 dt 加粗，dd 缩进 术语解释、FAQ、元数据键值对、配置参数说明 ❌ <table> 键值对型数据用 <dl>，表格数据用 <table>
<small> 附属说明 较小字号（约 80%） 版权声明、法律免责、表单提示文字 ❌ <span> 表示附属信息，非通用缩小文本
<sub> / <sup> 下标 / 上标 缩小并偏移 化学式（H2O）、数学指数（x2）、脚注引用[1] — 仅用于语义上下标，非视觉位移

强调层级建议

按视觉权重从强到弱排列：
最强 + color: #DC2626 严重警告、不可逆操作提示
强 （默认加粗） 核心结论、关键术语、必须遵守的规范
中 （斜体强调） 需要重读的词语、语义对比
弱 （高亮标记） 与用户当前上下文关联的内容

  .sidebar, .comments, .related-posts { display: none; }
  /* 正文全宽 */

  figure { page-break-inside: avoid; }

✅十四、文本内容展示 · 检查清单

✅ 每页是否只有一个 H1？
✅ 标题层级是否按 H1→H2→H3→H4 递进，无跳级？
✅ 正文字号是否在 15-17px 范围？
✅ 正文行高是否 ≥ 1.6？
✅ 段落间距是否使用 em 单位（相对字号）？
✅ 内容区最大宽度是否 ≤ 760px？
✅ 正文颜色是否不使用纯黑 #000？
✅ 正文段落是否设置了首行缩进 2em（长篇正文场景）？
✅ 标题/列表/引用后的首段是否取消缩进？
✅ 正文段落是否使用了两端对齐（text-align: justify）？
✅ 元数据区是否包含作者和日期？
✅ 图片是否设置了 width/height 防止 CLS？
✅ 图片是否使用了 loading="lazy"？
✅ 链接是否有下划线（不单独依赖颜色区分）？
✅ 代码块和行内代码是否样式明确区分？
✅ 引用块是否有左边框 + 背景色？
✅ 是否提供了打印样式（@media print）？
✅ 移动端字号和间距是否等比缩放？
✅ 关键警告/结论是否使用 <strong>（而非 <b>）？
✅ 句子中的重读强调是否使用 <em>（而非 <i>）？
✅ 键盘快捷键是否使用 <kbd> 标签？
✅ 专业缩写首次出现是否使用 <abbr title="...">？
✅ 文档修订（增删改）是否使用 <del>/<ins> 标记？
