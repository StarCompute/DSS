📋一、表设计规范

1.1 表命名规范

规则示例说明
小写 + 下划线 user_order 全小写英文，单词间用下划线分隔，禁止驼峰、中文、拼音
单数名词 user 而非 users 表名用单数形式，表示"用户表"而非"用户集合"
业务前缀 sys_ biz_ log_ 按模块加前缀区分：系统表/业务表/日志表/中间表
不超过 32 字符 sys_role_permission ✅ 避免过长的表名，控制在 32 字符内
禁止关键字 避免 order group key 不用数据库保留字作表名

1.3 表必须包含的结构字段

每张业务表必须包含以下标准字段：

字段名类型必填说明
id BIGINT 必须 主键，自增ID 或 雪花ID；禁用 UUID 字符串作主键
create_time DATETIME 必须 创建时间，默认 CURRENT_TIMESTAMP
update_time DATETIME 必须 更新时间，ON UPDATE CURRENT_TIMESTAMP
create_by BIGINT 推荐 创建人ID，记操作者
update_by BIGINT 推荐 更新人ID
is_deleted TINYINT(1) 推荐 逻辑删除标记：0=正常，1=已删除；禁止物理删除
version INT 可选 乐观锁版本号，并发更新场景必备
remark VARCHAR(500) 可选 备注说明

1.4 标准建表语句示例

CREATE TABLE biz_order (
    id          BIGINT       NOT NULL AUTO_INCREMENT  -- 主键ID
    order_no    VARCHAR(32)  NOT NULL                 -- 订单编号
    user_id     BIGINT       NOT NULL                 -- 用户ID
    total_amount DECIMAL(12,2) NOT NULL DEFAULT 0.00  -- 总金额
    status      TINYINT      NOT NULL DEFAULT 0       -- 状态: 0待支付 1已支付 2已取消
    create_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP
    update_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
    create_by   BIGINT       DEFAULT NULL
    is_deleted  TINYINT(1)   NOT NULL DEFAULT 0
    PRIMARY KEY (id),
    UNIQUE KEY uk_order_no (order_no),
    KEY idx_user_id (user_id),
    KEY idx_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单表';

1.5 表 & 字段必须加注释

每张表必须有 COMMENT；每个字段必须有 COMMENT；状态枚举字段必须注释每个值的含义。

⚠️ 反例： status TINYINT DEFAULT 0（无人知道 0/1/2 代表什么）
正例： status TINYINT DEFAULT 0 COMMENT '状态: 0待审核 1已通过 2已驳回 3已撤销'

2.3 关联字段命名规范

外键关联字段统一使用 {关联表名}_id 格式。

场景字段命名关联表示例
一对一{对方表}_id加唯一索引user_profile → user_id UNIQUE
一对多{一方表}_id加普通索引biz_order_item → order_id
多对多中间表含双方ID联合主键/联合索引role_permission → role_id + permission_id
树形结构parent_id自引用外键sys_dept → parent_id
多态关联{主体}_type + {主体}_id类型+ID组合source_type + source_id
💡 命名一致性原则： 当 user 表的主键是 id 时，所有引用 user 的外键必须统一叫 user_id，不可出现 uid / userid / creator 等变体。

2.4 字段类型选型规范

数据类型MySQL 类型适用场景注意事项
主键 BIGINT 所有表主键 不用 INT（易溢出）、禁用 UUID 字符串做主键
短字符串 VARCHAR(N) 名称、编号、标题等 N 按实际需求，不长于 500
长文本 TEXT / MEDIUMTEXT 描述、富文本、JSON 数据 不要给 TEXT 设默认值；不与 VARCHAR 混用
定长编码 CHAR(N) 固定长度如 MD5/身份证 N ≤ 64 且长度绝对固定才用 CHAR
整数 TINYINT / INT / BIGINT 数量、年龄、计数等 按值范围选型，不要一律 INT
金额 DECIMAL(M,D) 金额、费率、价格 严禁用 FLOAT/DOUBLE 存储金额！
日期时间 DATETIME 所有时间字段 不用 TIMESTAMP（2038 问题），统一 DATETIME
日期 DATE 纯日期（生日、统计日期） 只存日期不存时间的场景
布尔/状态 TINYINT(1) 开关、状态枚举、标志位 不用 ENUM（扩展性差），用 TINYINT + 注释
JSON JSON 扩展属性、配置、动态列 MySQL 5.7+ 原生支持；不替代关联表设计
二进制 BLOB / MEDIUMBLOB 文件存储 优先用对象存储（OSS/S3），数据库仅存 URL

2.6 是否为空 (NULL / NOT NULL) 规范

规则说明
尽量 NOT NULL，给 DEFAULT 除非业务上确实需要 NULL 表达"未知"，否则一律 NOT NULL + 默认值
主键 & 外键 必须 NOT NULL
字符串 NOT NULL DEFAULT ''，不用 NULL 表示"无"
数值 NOT NULL DEFAULT 0
时间 create_time/update_time 必须 NOT NULL；可选时间（如 expire_time）可 NULL
布尔/状态 NOT NULL DEFAULT 0（或含义明确的默认值）
⚠️ NULL 陷阱 NULL != ''、NULL != 0、COUNT(列) 不计 NULL、WHERE column != 1 不包含 NULL 行

📑三、索引设计规范

3.1 索引命名规范

索引类型命名前缀示例
主键PRIMARYPRIMARY KEY (id)
唯一索引uk_uk_order_no uk_user_mobile
普通索引idx_idx_user_id idx_create_time
联合索引idx_idx_status_time（字段按序拼接）
全文索引ft_ft_content

联合索引命名：idx_{字段1}_{字段2}_{字段3}，按索引列顺序拼接，一般不超过 3 个字段。

3.2 哪些字段必须建索引

场景建索引策略示例
主键自动聚簇索引id BIGINT PRIMARY KEY
外键普通索引，必须建所有 {xxx}_id 字段
WHERE 高频条件普通索引 / 联合索引status type user_id
ORDER BY 字段建立索引避免 filesortcreate_time sort_order
唯一约束唯一索引order_no mobile email
模糊查询（前缀）普通索引即可WHERE name LIKE '张%' — 可用索引
范围查询联合索引放最后idx_status_time (status, create_time)
覆盖索引SELECT 列均在索引中避免回表，性能最优

3.3 哪些情况不建议建索引

场景原因
区分度极低的字段如 is_deleted 仅 0/1 两值，索引无效
频繁增删改的字段索引维护代价大，写入性能下降
长文本字段TEXT/BLOB 不能直接建普通索引（需前缀索引）
表数据量很小< 1000 行，全表扫描更快
查询中很少使用的字段索引不是越多越好，占用空间、拖慢写入
作为表达式的字段WHERE YEAR(create_time) = 2025 不走索引
⚠️ 索引不是越多越好： 每多一个索引，INSERT/UPDATE/DELETE 就要多维护一个索引结构。一张表的索引数建议控制在 5 个以内。

3.4 联合索引最左前缀原则

联合索引 idx_a_b_c (a, b, c) 能覆盖的查询条件：

查询条件是否走索引原因
WHERE a = 1✅ 走命中最左列 a
WHERE a = 1 AND b = 2✅ 走命中 a + b
WHERE a = 1 AND b = 2 AND c = 3✅ 走全匹配
WHERE a = 1 AND c = 3⚠️ 只走 a中间跳过了 b，c 不走索引
WHERE b = 2❌ 不走不满足最左前缀
WHERE a = 1 ORDER BY b✅ 走排序也能利用索引顺序
WHERE a > 1 AND b = 2⚠️ 只走 a范围查询后的列不走索引
💡 联合索引列顺序建议： 区分度高的放前面 → 等值查询的放前面 → 范围查询的放最后。

3.5 常见建索引场景模板

业务场景索引方案
用户表查重 uk_mobile uk_email uk_username
订单列表查询 idx_user_id + idx_user_status_time (user_id, status, create_time)
日志按时间范围查 idx_create_time 或 idx_type_time (type, create_time)
树形结构查子级 idx_parent_id
模糊搜索（前缀） idx_name (name) — 支持 LIKE 'xxx%'
分页排序 idx_create_time 或联合 idx_status_time (status, create_time)
多对多中间表 PRIMARY KEY (a_id, b_id) + idx_b_id (b_id)

3.6 上线前必须 EXPLAIN 验证

所有线上 SQL 必须通过 EXPLAIN 检查执行计划：

关注指标目标值说明
typeconst / eq_ref / ref / range避免 ALL（全表扫描）、index（全索引扫描）
key命中预期索引名不能为 NULL（未走索引）
rows尽量小预估扫描行数越少越好
Extra避免 Using filesort Using temporary出现则说明需要优化

✍️四、SQL 编写规范

4.1 查询规范

规则正例 ✅反例 ❌
禁止 SELECT * SELECT id, name, status SELECT *
避免函数作用索引列 WHERE create_time >= '2025-01-01' WHERE DATE(create_time) = '2025-01-01'
避免隐式类型转换 WHERE user_id = 123 WHERE user_id = '123'（列是 BIGINT）
IN 不超过 500 个 分批次或用临时表 JOIN WHERE id IN (1,2,3,...5000)
分页必须带 ORDER BY ORDER BY id LIMIT 0,20 LIMIT 0,20
大偏移量用游标分页 WHERE id > 10000 LIMIT 20 LIMIT 10000, 20

4.2 写入规范

规则说明
必须显式指定字段INSERT INTO t (a,b,c) VALUES (...)，不能省字段列表
批量写入用批量 INSERT 替代逐条插入；每批建议 500~1000 条
更新带版本号UPDATE ... SET ... version = version + 1 WHERE id = ? AND version = ?
逻辑删除UPDATE t SET is_deleted = 1 WHERE id = ?，禁止物理 DELETE
事务尽量短事务内不要做 RPC 调用、文件 IO、外部 API 请求

✅七、数据库设计检查清单

  • 表名使用小写+下划线，单数名词
  • 所有表包含 id、create_time、update_time 标准字段
  • 字段类型和长度选择合理，避免过度使用 VARCHAR(255)
  • 关联字段命名规范（如 user_id 关联 user 表）
  • 高频查询条件字段已建立索引
  • 避免 SELECT *，明确列出所需字段
  • SQL关键字统一大写，表名/字段名统一小写
  • 字符集统一使用 utf8mb4
